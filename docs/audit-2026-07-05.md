# Demand-tool audit — 5 July 2026

A full sweep of the code (~30,000 lines: every data query, all 76 API routes, the
dashboard, the capture screen, and settings) looking for **bugs** and
**performance problems**. Every finding below was found by one reviewer and then
independently re-checked by a second reviewer whose job was to *disprove* it — so
what's listed here survived a deliberate attempt to knock it down. A handful of
first-pass findings did not survive and were dropped (see the end).

**How to read this**

- **Part 1 — Bugs** are *reported only, not changed.* Fixing a bug changes what
  the tool shows or does, which you asked to decide case-by-case. Nothing in
  Part 1 has been touched.
- **Part 2 — Performance.** The items marked **✅ Applied** are on a separate
  `perf-audit` branch (a preview copy, not live). They make the tool do less work
  but produce the *exact same* screens, numbers, and files — verified. The items
  marked **Proposed** are real wins but carry enough rewrite risk that I left them
  for you to schedule rather than apply blind.

Severity is High / Medium / Low by real-world impact, not by how hard it is to fix.

## Part 1 — Bugs (reported, not changed)

### The biggest one: another study can be edited through your study's link

**Severity: High.** This is a cluster of ~15 places, same root cause.

Every study is reached by its 6-character code (e.g. `MVJN2A`), and that's
deliberate — there are no user accounts. The problem: about fifteen "save" and
"delete" actions check only that *some* study matches the code in the address
bar, then change the row by its internal id **without checking that row belongs
to that study.** Those internal ids are handed out in every study's data,
exports, and templates, so they aren't secret once a code has been shared.

Concretely, someone holding *any* valid study code could, if they obtained a row
id from another study, rename or delete that other study's categories, or edit
and even delete another study's captured entries. The routes that *do* check
correctly (milestones, sub-questions, cases, the lifecycle classifiers) show the
intended pattern; these older ones were never brought in line.

Affected: the PATCH/DELETE handlers for contact methods, demand types, handling
types, life problems, lifecycle stages, points of transaction, system conditions,
thinkings, value steps, what-matters types, work sources, work-step types, work
types; the entry edit/delete route (`entries/[entryId]`); the entry lifecycle
override; and the work-step "promote from cluster" action (which trusts a list of
block ids sent by the browser).

**Why it's not auto-fixed:** the fix makes these actions start *rejecting*
cross-study requests — a behaviour change. I've put a task on your dashboard for
it. It's the one item here I'd genuinely prioritise, even though there's no sign
it has been exploited.

### Dates can land on the wrong day (timezone bugs)

**Severity: Medium.** Several places compute "today" or a chosen date in UTC while
the screen shows the *local* day. For you in Denmark (UTC+1/+2), between local
midnight and 01:00–02:00 the two disagree by a day.

- **Capture backdates a touch by a day (the one that matters most).** The date a
  new work touch defaults to is computed in UTC. Because a flow touch's "effective
  date" drives the capability/XmR charts, a touch entered late at night can be
  silently filed under yesterday — moving where it sits on the very charts this
  tool exists to produce. Same UTC/local mismatch in the edit modal's date field
  and the milestone "completed on" default. (The decision-box answer dates already
  guard against this by pinning to noon UTC — the capture defaults just don't.)
- **Custom date range on the dashboard** starts the range at UTC midnight while
  ending it at local end-of-day, so the *first* 1–2 hours of the chosen start day
  are dropped from every chart and export using a custom range. (The end of the
  range is fine — only the start drifts.)

### Failed saves that look like they worked

**Severity: Medium.**

- **Edit-entry modal.** If a save or delete fails on the server, the modal closes
  as if it succeeded and the list refreshes to the old data — the collector
  believes their edit stuck when it didn't. (The main capture form does check for
  failure; the modal doesn't.)
- **A dropped network connection freezes buttons.** In several places a save runs
  without a safety net, so a flaky-wifi failure leaves the button spinning /
  disabled until the page is reloaded — and the typed text is lost. Affects the
  main capture Save, the "add a category" box, the edit modal, the case context
  save, and the milestone save/close. (Some paths already handle this; these don't.)

### Export → re-import loses data

**Severity: Medium.**

- **"What matters" category is dropped on round-trip.** The export writes the
  column header as *"What Matters Categories"* (plural) but the importer looks for
  *"What Matters Category"* (singular), so re-importing an exported file silently
  loses that column on every row.
- **A bad date in an imported file can wipe a touch's work steps.** When editing
  an entry, the existing work blocks are deleted first and then re-inserted; an
  unparseable date string makes the re-insert fail *after* the delete, so the
  steps are gone and the request errors.
- **Import that fails partway leaves half the rows in and reports nothing.** If a
  row fails mid-import, the whole request errors after earlier rows were already
  saved, and the browser gets neither the success count nor the error list — so
  the natural "try again" duplicates everything that already went in.
- **"Download everything" keeps only the first system condition per work block.**
  Since blocks can carry several conditions, the full-export sheet drops all but
  the first on multi-condition blocks (the edit screen shows them all correctly).

### Ordering / reorder glitches in settings

**Severity: Medium.** Rooted in one thing: when you delete an item from a list the
remaining items aren't renumbered, and adding a new item reuses a number that can
already be taken — so two items end up with the same sort position.

- **Reorder arrows go dead.** Once two sub-questions (or value steps) share a
  position number, the ↑/↓ buttons appear to do nothing, and their order on reload
  is whatever the database happens to return.
- **Value-step move doesn't fully stick.** A move renumbers the list on screen but
  saves only the two swapped rows, so the saved order can differ from what you see
  until the next reload.

### Templates quietly lose the newer decision-box settings

**Severity: Medium.** "Save as template" snapshots an *older, now-unused* table and
skips the live ones added since. A study created from such a template silently
loses: percent formatting on calculated fields (an 80% field shows as `0.8`), the
per-question and per-milestone "applies to which demand types" rules, and the
"not mandatory" settings. Completion and auto-close then behave differently from
the study the template was made from, with no warning.

### Smaller bugs

**Severity: Low** (grouped; each is real but narrow):

- **Editing an entry can leave stale hidden data.** Reclassifying in the edit
  modal clears less than the capture form does, so a failure→value change can keep
  an "original value demand" link and a failure cause attached; the modal's
  work-type list also isn't filtered by classification, so you can pick a failure
  work type for a value entry (impossible on the capture screen).
- **Undo leaves the deleted touch in some lists** until the next save/reload
  (clicking Edit on it then errors).
- **Re-tagging a work step to "Value" can keep invisible conditions attached** in
  one of the two editing modes (the other mode clears them).
- **A blank line typed inside one work step** splits it into two on the flow rail,
  the second wrongly coloured green (value).
- **Rapid clicks can double-save or show the wrong data:** pressing Enter during a
  slow save can post a touch twice; quickly switching customers can briefly show
  one customer's board while saves attach to another; fast filter/toggle changes on
  the dashboard can leave charts showing a previous filter; opening "examples" for
  two categories quickly can show one under the other.
- **`openedAt: null` sets a case's open date to 1970** (epoch) instead of being
  rejected — which then distorts lead-time charts.
- **A malformed number in a formula** (e.g. `1.2.3`) is silently accepted as `1.2`
  rather than flagged; an invalid formula (e.g. `2 +`) can be saved and then shows
  blank forever on the capture screen; and the formula builder offers text/date
  fields as numeric inputs, producing formulas that can never compute.
- **Clearing an "as soon as possible" anchor doesn't stick** for older
  what-matters rows (it snaps back to the previous milestone).
- **Renaming an answer option** briefly detaches its follow-up questions on screen
  until the page is reloaded (the server handles it correctly; only the on-screen
  copy goes stale).
- **A bad `from`/`to` value in a dashboard link** returns a server error instead of
  a clean "bad request".
- **The settings PIN is cosmetic:** it's sent to every browser that loads the
  study and checked on the client, so anyone with the code can read it. Fine if you
  only meant it as a speed-bump; not a real lock.
- **The page always reports its language as English** to screen readers even in
  Danish/Swedish/German.
- **The info (ⓘ) tooltip** stays put when you scroll, drifting away from what it
  explains.
- **Duplicate answer-option labels** aren't prevented and confuse the branching
  view.
- **Three dev-only seed scripts are broken** (call a removed endpoint / import a
  removed table) and the migration-apply script splits SQL naively — harmless
  today because every migration so far is plain DDL, but a sharp edge for a future
  migration containing a `;` inside text or a `DO $$…$$` block. This is the script
  you paste against **production**, so worth knowing.

## Part 2 — Performance

All "Applied" items were checked to produce byte-for-byte identical output (same
screens, same numbers, same exported files) — they only remove wasted work.

### ✅ Applied (on the `perf-audit` preview branch)

- **Saving a touch is much faster.** Saving used to write each work block and each
  condition/what-matters/thinking link to the database one at a time; on a rich
  touch that's 30–50+ round-trips. Now each group is written in a single batched
  insert. Biggest single win, on the screen where speed matters most.
- **Opening the edit modal no longer downloads the whole study.** It used to pull
  every entry in the study to find the one you clicked; now it fetches just that
  one (also naturally scopes it to the study).
- **Independent database reads now run in parallel** instead of one-after-another,
  in: the dashboard data build, the entries list + today-count, the case-context
  recompute (matched to the faster pattern already used elsewhere), the export's
  category lookups, and the lifecycle classifier (which also stops re-querying data
  it already had).
- **Charts stop rebuilding on every keystroke.** On the dashboard, the date range
  was recomputed on every re-render, which re-triggered every chart's data fetch
  while you typed or hovered; the capability chart and the synthesis charts
  rebuilt their data on every keystroke in a note/rename field. These now reuse
  their data unless the inputs actually change.
- **The decision box does less repeated work** while you type an answer (it was
  rebuilding its whole question tree and doing an unnecessarily slow lookup on
  every character).
- **Settings stops sending pointless saves** — tabbing through a field without
  changing it no longer fires a save.

### Proposed (real wins, left for you to schedule)

These are genuine and were verified as output-identical, but each is a larger
rewrite with more room to slip, so I didn't apply them blind:

- **The capture board re-renders fully on every keystroke** — every saved touch
  card re-parses itself as you type in the composer. The fix (splitting the touch
  card into its own memoized piece) is the highest-value remaining perf item but
  touches the busiest screen, so it deserves a careful, quiet-window change.
- **The settings page re-renders fully on every keystroke** in any field — same
  shape of fix.
- **The dashboard recomputes all its chart data on any toggle** (including the
  word-cloud). Cleaning this up needs the page split into smaller pieces.
- **Two list-refetch-after-tiny-change spots** (case list after every save; full
  entries refetch after a single edit) could fetch far less — but the safe version
  is *behaviour-sensitive* (it changes when data is considered fresh), so it needs
  your call rather than a silent swap.
- **A couple of database queries could be tightened** with raw SQL (the case list
  runs the same lookup twice per case; two reorder operations do one write per
  row). Modest wins, and raw-SQL rewrites are exactly where a subtle
  not-quite-identical result could sneak in, so I left them as reported.
- **The formula builder saves after every button click** (each an intermediate,
  often invalid, version). Saving only when you leave the field would be cleaner,
  but the change shifts *when* the save happens, so I left it for you rather than
  risk a rare unsaved-formula edge on the live tool.

### Noted, not acted on

- **`getReclassificationCount` is dead code** — it loads every entry just to count
  them, but nothing calls it. Safe to delete.
- **The "entries today" counter is computed but never shown** anywhere — likewise
  safe to remove if it's not planned for.

## First-pass findings that did **not** survive re-checking

Listed for transparency — these were raised then disproven:

- A missing `duration_months` branch in the "did we deliver?" logic — **not a
  bug**: that field type is deliberately not linkable, so the branch is
  unreachable.
- The clustering routine's O(n²) comparison — **intentional** and documented; no
  output-identical faster version exists at this scale.
- A suspected month-boundary error in formula date math — real in principle but
  **cannot occur** at Denmark/Sweden's timezone offsets.
- A dashboard chart "ignoring" the top-level filter — it has its **own** scope
  control; this is a design choice, flagged for you rather than fixed.
